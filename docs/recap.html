Internal research recap · self-improving security validation

Loupe → PenPal: can a vuln-finding agent self-evolve at telling real from benign?

Open-source proof-of-concept · 13 experiments · no fine-tuning · DeepSeek-V4-Pro via Together AI · as of 2026-06-22

The problem. PenPal finds vulnerabilities; security engineers can't verify each one deterministically. The painful class is the benign positive — a finding that is genuinely a bug in the code but is neutralized in deployment (a WAF, an auth gateway, an unreachable route, a disabled flag). They erode trust and burn triage time.

The thesis. When findings get false-positive feedback, the agent should learn the context behind the FP and reuse it to inform downstream decisions — self-evolution without fine-tuning, via memory + grounded verification.

The one-line result. The FP problem splits in two. Code-level FPs → the lever is context (give the model the dataflow; no learning needed). Deployment-level FPs → the lever is self-evolution (memory of shared environmental facts) — it cuts FPs ~80%… but the same memory-trust erodes safety (stale or leaky context causes false-negatives), so it must annotate, not silently suppress.

The sharper result (E14). Of the two ways to self-evolve, the durable one is learning reproduction technique (a procedure — never goes stale, always helps), gated by reproduction as the eval (an agent either reproduces the finding or it doesn't — unfakeable). The fragile one is learning "this is benign" as a heuristic — it over-suppresses real bugs and must instead be grounded in an actual reproduction attempt. Net: evolve a technique library; verify by sandboxed reproduction; annotate what fails to reproduce.

1 · The premise you bring — and why it matters here

From detection engineering: vulnerabilities fall into groupings — by weakness class (CWE), by sink pattern, by the deployment shape they live in — and in a large enterprise codebase those groupings can be categorized and learned.

This is precisely the precondition our central positive result needs. Self-evolving memory only pays off when a false positive's cause is a reusable, shared fact (E11). If FPs cluster by CWE × deployment-context — as your experience suggests — the memory transfers across the cluster and compounds. If they were all one-offs, memory would be worthless and only per-finding context would help. Whether they cluster at enterprise scale is the single biggest open question (see Gaps §5).

2 · The loop we built (and the field's framing of it)

Everything maps onto one governed cycle — the same loop the current "self-evolving agents" literature describes:

observe → evaluate → propose improvement → verify → commit  ↻

What evolves (risk ladder)

Prompt — safest. Done (real DSPy/GEPA, E4/E10).

Memory / context — our main lever. Done (E1–E13).

Code — needs sandbox. Not pursued (our evidence says info, not procedure, is the bottleneck).

Weights — highest risk. Excluded by design (no fine-tuning).

The two governance laws we tested

Goodhart: the evaluation function must sit where the agent can't touch it. We violated this twice and watched evolution game the metric (E9, E10).

Safety is eroded, not added: the trust that yields the gain is the same trust that produces false-negatives under drift/leak (E12, E13). Measured, not assumed.

5 · What we've learned (the durable findings)

① The FP problem partitions cleanly

Code-level (sanitizer in the method) → fix with context (E10), no learning. Deployment-level (WAF/gateway/flag) → fix with memory (E11), context can't. Knowing which one a finding is tells you which tool to reach for.

② Self-evolution needs a shared cause

Memory pays off only when the FP's cause recurs (E11). This is where your CWE-grouping premise is load-bearing. One-off FPs don't transfer; clustered ones compound.

③ The gain and the danger are one mechanism

The trust that cuts FPs 80% drives FNs to 100% under drift (E12) and −22% recall under leaky controls (E13). You cannot get the upside without governing the downside.

④ Capability amplifies the risk

The stronger model trusts grounded memory more, so it over-suppresses more (E12/E13). Better models need more re-verification discipline, not less.

⑤ Goodhart is real and fast

We watched evolution game a misspecified metric twice, in opposite directions — "never submit" (E9) and "always flag" (E10). The eval/oracle must be architecturally untouchable.

⑥ The safe product form is annotate, not suppress

Given ③/④, memory should enrich the finding with grounded, fresh evidence + confidence and let downstream decide — never silently drop it (flag-don't-flip, E3 at product scale).

⑦ Two mechanisms — one durable, one fragile

Fact-memory (E11: "service X is benign") is high-gain but goes stale & over-suppresses (E12/E13). Procedural skill (E9/E14: "how to reproduce / verify") is a procedure — it doesn't go stale and only ever helps. Bet on the procedural line.

⑧ "Benign" must be grounded in execution

E14: technique-learning is safe & monotonic, but a learned "benign" rule over-suppresses and is (correctly) rejected by the guard. The benign verdict has to come from a real reproduction attempt that fires or doesn't — not a heuristic. Reproduction = a Goodhart-safe oracle.

⑨ Grounded reproduction unifies both levers

E15: actually attempting the exploit naturally folds in code-context (E10 — a sanitizer makes it fail) and deployment-context (E11 — a WAF 403s it). On a real oracle: capability 1.00, 0 false claims, 0 over-suppression. It turns over-suppression into a measurable coverage/effort problem → confidence = how hard we tried.

7 · Where we go next

The unlock is one artifact, not one experiment. Instrument PenPal to capture structured, grounded FP feedback — finding + human verdict + reason + evidence — as a labeled corpus. That single dataset simultaneously: (1) lets us measure self-improvement honestly via the matched-pairs gain metric, (2) answers the clustering question (§5②) that decides whether self-evolution is even the right tool, and (3) bootstraps the automated grounded verifier — with humans as the drift audit.

Concretely, in order: (a) measure FP clustering on real findings → (b) matched-pairs gain at scale on real code → (c) ship annotate-not-suppress enrichment → (d) automate grounded checks + set re-verification cadence from observed drift.

Already done (Set 3): the execution-grounded reproduction now runs on real Cybench with zero self-deception — that build is complete. The two remaining, both pointed to by the benchmark phase: (i) a cross-function slice arm to rescue the Set-2 / trust-boundary cases (give the model the reachable dataflow, not an isolated function); (ii) CVE-Bench/UIUC for the first real-CVE deployment-context benign-positive (an exploit that doesn't fire in the sandbox) — the project's original motivation, now within reach.
